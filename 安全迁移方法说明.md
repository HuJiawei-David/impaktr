# 安全迁移方法说明 - 不会重置数据库

## ✅ 最安全的方法（推荐）

### 方法 1: 手动执行 SQL（最安全，推荐）

**为什么最安全：**
- ✅ 只添加新的枚举值，不修改现有数据
- ✅ 使用 `IF NOT EXISTS`，即使重复运行也不会出错
- ✅ 不会删除任何现有数据
- ✅ 不会影响现有的通知记录
- ✅ 完全由你控制，不会意外执行其他操作

**操作步骤：**
1. 登录你的数据库控制台（Neon、AWS RDS 等）
2. 打开 SQL Editor
3. 复制粘贴以下 SQL：

```sql
ALTER TYPE "NotificationType" ADD VALUE IF NOT EXISTS 'CERTIFICATE_ISSUED';
ALTER TYPE "NotificationType" ADD VALUE IF NOT EXISTS 'RANK_UP';
ALTER TYPE "NotificationType" ADD VALUE IF NOT EXISTS 'EVENT_JOINED';
ALTER TYPE "NotificationType" ADD VALUE IF NOT EXISTS 'MONTHLY_REPORT';
```

4. 点击执行
5. 验证（可选）：

```sql
SELECT enumlabel 
FROM pg_enum 
WHERE enumtypid = 'NotificationType'::regtype 
ORDER BY enumlabel;
```

**优点：**
- 100% 安全，不会重置数据库
- 可以立即看到结果
- 不会影响其他表或数据

### 方法 2: 使用迁移脚本（安全）

**为什么安全：**
- ✅ 使用 `IF NOT EXISTS`，重复运行安全
- ✅ 只添加枚举值，不删除数据
- ✅ 有错误处理和验证

**操作步骤：**
```bash
# 确保 DATABASE_URL 已设置
node apply-notification-enum-migration.js
```

**优点：**
- 自动化，不需要手动输入 SQL
- 包含验证步骤
- 有详细的日志输出

---

### 方法 3: Prisma Migrate Deploy（安全，用于生产环境）

**为什么安全：**
- ✅ 只应用待执行的迁移，不会重置数据库
- ✅ 不会修改现有数据
- ✅ 适用于生产环境

**操作步骤：**
```bash
npx prisma migrate deploy
```

**注意：**
- ⚠️ 只用于生产环境
- ⚠️ 不要使用 `prisma migrate dev`（开发环境可能重置）

**优点：**
- 正式的生产环境迁移流程
- 跟踪迁移历史
- 可以回滚（如果需要）

---

## ❌ 不安全的方法（避免）

### ❌ `prisma migrate dev`（开发环境）
- 可能创建新的迁移
- 在开发环境可能重置数据库
- **不要在生产环境使用**

### ❌ `prisma db push`（开发环境）
- 可能重置开发数据库
- **不要在生产环境使用**

### ❌ `prisma migrate reset`（任何环境）
- 会**删除所有数据**并重置数据库
- **绝对不要使用**

## 🔒 为什么这些方法是安全的？

### SQL 操作说明

```sql
ALTER TYPE "NotificationType" ADD VALUE IF NOT EXISTS 'CERTIFICATE_ISSUED';
```

这个 SQL 语句：
- **只做一件事**：在枚举类型中添加一个新值
- **不删除**：不会删除任何现有的枚举值
- **不修改**：不会修改现有的枚举值
- **不删除数据**：不会删除任何表中的数据
- **不影响现有记录**：所有现有的通知记录保持不变

### 为什么 `IF NOT EXISTS` 很重要？

- 如果值已经存在，不会报错
- 可以安全地重复运行
- 不会因为重复执行而失败
